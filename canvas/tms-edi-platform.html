Canonical Data Model — Hub-and-Spoke Pattern
Why Canonical Model?
Without canonical model: N partners × M targets = N×M mappings
With canonical model: N partners + M targets = N+M mappings

Example: 10 partners, 5 TMS targets
Without: 50 mapping configurations
With: 15 mapping configurations
Partner_A (XML) ──┐
Partner_B (CSV) ──┤
Partner_C (X12) ──┼──→ CANONICAL → TMS_Tables
Partner_D (JSON) ─┤

Processing Workflow
End-to-end pipeline from file receipt to TMS staging table insertion. Each step is independently monitored, retry-capable, and emits Kafka events for downstream consumers.
Main Processing Pipeline
01
File Receipt EDI Gateway
File arrives via SFTP/FTP/REST/AS2/Email. Gateway authenticates partner via API key or certificate. Partner code identified from path or header. Event: file.received
02
Validation & Storage File Receiver
Compute SHA-256 checksum. Check for duplicates via checksum index. Validate file size limits. Store raw file to MinIO/S3. Insert record into TMS_FILES with status=RECEIVED.
03
Format Detection & Analysis Structure Analyzer
Auto-detect file format from extension + magic bytes. Parse file. Generate hierarchical schema tree with full field paths. Cache schema in Redis for mapping UI. Event: file.analyzed
04
Mapping Template Lookup Mapping Engine
Query MAPPING_HEADER for active mapping matching (partner_id + file_type). If no mapping exists and mode=AUTO → trigger AI auto-mapping suggestions. If mode=MANUAL → halt and notify operator.
05
Rule Execution & Transformation
Apply mapping rules in sequence order. Execute transformations: DATE_FORMAT, UPPER, CONCAT, IF/ELSE, LOOKUP. Handle defaults and conditionals. Build canonical data structure in memory.
06
Canonical Validation Validation Engine
Validate mandatory fields (external_order_id, customer_code, order_date). Check data types, formats, business rules. Check for duplicate order IDs per partner. Log all validation errors to EDI_ERROR_LOG.
07
Staging Table Insertion Processing Engine
Transactional insert into EDI_ORDER_HEADER, EDI_ORDER_LINES, EDI_CARGO_DETAILS, EDI_COSTS. Use batch inserts for performance (configurable batch size). Update TMS_FILES status=PROCESSED.
08
TMS Integration
Post-processing hook triggers TMS import procedures. Canonical data promoted to live TMS order tables. Confirmation event sent back to partner if configured. Status updated, metrics published to Prometheus.
Processing Modes
⚡
AUTO Mode
Zero-touch processing
processing_mode = AUTO
File arrives → immediately consumed by Kafka consumer → full pipeline executed automatically → status updated. No human intervention required. Ideal for trusted partners with stable formats.
👆
MANUAL Mode
Operator-triggered processing
processing_mode = MANUAL
File stored as PENDING. Appears in File List screen awaiting operator action. Operator reviews file in viewer, confirms mapping, then clicks "Process" button → triggers pipeline via POST /api/files/{id}/process.
🕐
SCHEDULED Mode
Cron-based batch processing
processing_mode = SCHEDULED
Files accumulate during the day. Quartz scheduler triggers batch job at configured cron expression (e.g., 0 0 2 * * ? = 2AM daily). Spring Batch processes all pending files in optimized bulk.
Retry Logic
On processing failure, the system:
1. Logs error to EDI_ERROR_LOG with full context
2. Increments retry_count on TMS_FILES
3. Schedules retry with exponential backoff (1m → 5m → 15m → 1h)
4. After max_retries (configurable, default 3) → status = ERROR
5. Dead Letter Queue (DLQ) in Kafka holds failed messages
6. Alert sent to monitoring dashboard
7. Operator can manually retry from Error Viewer UI

File Ingestion Module
Multi-channel file ingestion supporting SFTP polling, FTP, REST API push, AS2, and email. Each channel is independently configured per partner with automatic partner detection.
Ingestion Channels
📡
SFTP Polling
Spring Integration SFTP Adapter
Most Common
Polls partner SFTP servers on configurable schedule (cron). Moves processed files to /archive folder. Supports key-based and password authentication. Detects partner from folder path pattern: /sftp/partners/{PARTNER_CODE}/inbox/
sftp.host=sftp.partner.com sftp.port=22 sftp.user=ediuser sftp.private-key=/keys/partner_rsa sftp.remote-dir=/inbox sftp.poll-interval=60000
🌐
REST API Push
Webhook / HTTP Push
Partners POST files to our endpoint. Authentication via API key header X-API-Key or JWT. Supports multipart file upload and Base64 encoded content. Rate limited per partner. Async processing via Kafka.
📧
AS2 Protocol
EDI Standard Protocol
Enterprise EDI
Industry-standard AS2 protocol for X12 and EDIFACT. Supports MDN acknowledgements, signing, and encryption. Partner identified from AS2-From header. Uses Mendelson AS2 or Boomi AS2 connector.
📨
Email Attachments
IMAP / POP3 Polling
Spring Integration IMAP adapter polls mailbox for EDI attachments. Filters by sender address mapped to partner_id. Extracts attachments and routes to processing pipeline. Replies with acknowledgement email.

AI Auto Mapping Assistant
Machine learning-powered mapping suggestions using field name similarity (cosine similarity on embeddings), historical mapping patterns, and data type analysis. Reduces mapping setup time from hours to minutes.
How AI Mapping Works
1
File Structure Extraction
Schema tree generated from analyzed file. All field paths with data types and sample values extracted into feature vectors.
2
Semantic Similarity Scoring
Field names tokenized and embedded using a fine-tuned transformer model. Cosine similarity computed against canonical field embeddings. Top-N candidates ranked per target field.
3
Historical Pattern Matching
Query MAPPING_LINES table for similar field paths mapped by same or similar partners. Boosts confidence scores for historically confirmed mappings.
4
Data Type Validation
Check that suggested source field type is compatible with target canonical type. DATE → DATE, NUMBER → NUMBER. Cross-type mappings flagged with warning.
5
Suggestion Presentation
Suggestions displayed in mapping designer with confidence % and source. User accepts, modifies, or rejects. Accepted suggestions fed back into training data for continuous improvement.

Error Recovery Workflow
!
Error Detected
Exception caught in processing pipeline. Stack trace captured. Entry_no and mapping_line_id linked to error record. TMS_FILES.status = ERROR.
2
Error Logging
Insert into EDI_ERROR_LOG with error_type, error_code, field_path, full message, and timestamp. Publish to Kafka error topic for monitoring consumer.
3
Auto-Retry (if applicable)
Transient errors (DB failures, timeouts) queued for retry. Exponential backoff: 1 min → 5 min → 15 min → 1 hour. After max_retries → permanent ERROR state.

Security Controls
Transport Security
All external connections over TLS 1.3. SFTP uses public key authentication. AS2 uses certificate-based signing. Internal service mesh uses mTLS (Istio). No plain-text credentials stored — HashiCorp Vault.
Data Security
File blobs encrypted at rest (AES-256). Database TDE enabled. PII fields (addresses, customer data) optionally encrypted with per-partner keys. GDPR-compliant data retention with auto-purge policies.
Network Security
WAF (Web Application Firewall) on API Gateway. Rate limiting per partner (configurable). IP allowlist support for SFTP partners. DDoS protection via cloud provider. VPC isolation for internal services.
Compliance & Audit
SOC 2 Type II ready. Complete immutable audit trail in append-only audit table. All file downloads, mapping changes, user logins, and processing triggers logged. Configurable retention (default: 7 years for EDI audit).
